🎯 Vấn đề chúng ta muốn giải quyết

Context & motivation — tại sao chúng ta cần FCI BMad Kit

📁 Tài liệu rải rác

PRD/Story lưu mỗi nơi một chỗ — không ai biết bản nào mới nhất. Người mới join mất hàng giờ tìm context trước khi làm việc được.

🔄 Context mất khi handoff

Dev nhận story thiếu context → hỏi ngược lại BA/PO nhiều lần → delay cả sprint. Không có format chuẩn nên mỗi người handoff một kiểu.

🤖 Mỗi người dùng AI khác nhau

Không có prompt chuẩn → chất lượng output dao động lớn giữa các thành viên. Không reproduce được — người này làm tốt, người kia ra kết quả khác hẳn.

🗂️ Đa repo phức tạp

Feature đụng 2–3 repo (octavia, octavia-dashboard...) — không ai biết thay đổi ảnh hưởng đến đâu. Dev phải tự scan code thủ công, dễ miss.

💡 FCI BMad Kit giải quyết tất cả: agent chuẩn hóa cho từng vai, luồng handoff rõ ràng qua file Git — mỗi người làm việc trên máy mình, không cần server chung.

Customization Layering

BMad cung cấp hệ thống 3-layer override — FCI dùng đúng kiến trúc này để customize agent:

customize.toml Kit defaults

Cấu hình gốc của kit — baked vào source code FCI

_bmad/custom/fci-po.toml Team · git commit

Override cho toàn team — commit vào repo

_bmad/custom/fci-po.user.toml Personal · gitignore

Override cá nhân — không commit, không ảnh hưởng team

Quy tắc merge: Scalar → ghi đè · Array → append · Array-of-tables → merge theo key code
Kiến trúc của BMad — FCI dùng đúng như thiết kế, không hack.

FCI dispatch như thế nào?

Khi bạn chọn menu trong agent FCI, agent không tự xử lý mà dispatch sang skill BMad tương ứng. FCI chỉ thêm context (repo, handoff path) trước khi gọi.

Activation Flow — xảy ra mỗi lần gọi agent

1

Hỏi pull từ repo docs

Agent tự chạy git pull — user chỉ gõ "có" hoặc "không". Không cần nhớ lệnh git.

2

Đọc workspace.yml + list_repos

Biết dự án gồm những repo nào, URL clone của từng repo.

3

Auto-detect repo liên quan

Đọc handoff/story để biết feature cần đụng repo nào — không hỏi khi đã rõ.

4

Làm việc → Hỏi push

Sau khi xong, agent tự git add + commit + push. Nếu bị reject, tự rebase rồi push lại.

PO/BA không cần biết git — agent làm hết. Bạn chỉ trả lời có / không.

🔍 ④ Codegraph MCP — đọc code thông minh

FCI Custom — tích hợp trực tiếp vào activation flow; không phải addon tùy chọn

Là gì?

@optave/codegraph

Graph-index toàn bộ codebase → khi query trả về symbol / dependency / impact thay vì đọc raw file. Tiết kiệm token và nhanh hơn nhiều so với đọc file thủ công.

Ai dùng?

📊 BA (Vanh)

Hiểu constraint kỹ thuật trước khi viết story → story thực tế hơn, ít hỏi ngược lại Dev.

⚙️ Dev (Zitech) — bắt buộc

Chạy menu AN trước khi đụng code → biết blast radius, tránh phá module khác.

🧪 Tester (Hanh)

Navigate codebase khi viết test plan → biết chỗ nào cần cover edge case.

Cách dùng

// 1 MCP server phục vụ nhiều repo // --multi-repo flag khi start server // Xem repo đã đăng ký: codegraph.list_repos() // Lấy context của repo: codegraph.context({ repo: "octavia" }) // Tìm impact của một symbol: codegraph.impact({ repo: "octavia-dashboard", symbol: "HealthCheck" }) // Tìm định nghĩa symbol: codegraph.definition({ repo: "octavia", symbol: "WeightCalculator" })
⚠️ Quan trọng — đã baked vào agent:
Codegraph rất tốt để LOCATE (tìm symbol, file, import chain). Nhưng call-graph (impact) có thể thiếu edge trên Python vì dynamic typing (octavia dùng Python).

Agent được hướng dẫn đối chiếu với grep/ripgrep trước khi kết luận blast radius — không tin tuyệt đối vào call-graph một mình.

Node 22 requirement

Codegraph dùng better-sqlite3 với prebuilt binary — cần Node.js 22. Script cài đặt tự kiểm tra và báo lỗi nếu version không đúng.

🚀 Cài đặt — 1 lệnh, dùng ngay

Không cần config thủ công — script tự cài BMad + Codegraph + 4 agent

Single-repo (project riêng lẻ)

# Trong thư mục project của bạn: curl -fsSL \ https://raw.githubusercontent.com/\ vncharles/fci-bmad-kit/main/install.sh \ | bash # Hoặc với custom options: USER_NAME="Trọng" COMM_LANG="Vietnamese" \ ./install.sh
Script cài: BMad + module fci + Codegraph MCP + 4 agent. Không cần clone thủ công.

Sau khi cài, gọi ngay trong Claude Code:

/fci-po   → Thanh (Product Owner) /fci-ba   → Vanh (Business Analyst) /fci-dev  → Zitech (Developer) /fci-tester → Hanh (QA Tester)

Multi-repo — Team mode

# Trong workspace trống, chọn role: ROLE=po ./init-workspace.sh ROLE=ba ./init-workspace.sh ROLE=dev ./init-workspace.sh ROLE=tester ./init-workspace.sh # Script hỏi URL repo docs → # đọc workspace.yml → clone app repo → # build codegraph → cài BMad → push
Người đầu tiên khai báo repo → push workspace.yml → người sau chỉ paste URL docs repo.

workspace.yml — registry chung

project: lbaas docs_repo: git@gitlab.fci.vn:fci/docs.git repos: - name: octavia url: git@gitlab.fci.vn:fci/octavia.git roles: [ba, dev, tester] - name: octavia-dashboard url: git@gitlab.fci.vn:fci/octavia-dashboard.git roles: [ba, dev, tester]

Lưu ý vận hành

💬 Thông báo thủ công

Git không tự ping người khác — sau mỗi push cần tự nhắn Slack/GitLab cho người nhận pull.

🔄 Rebuild Codegraph

Khi code thay đổi nhiều, chạy: codegraph build <repo-name> để index lại. Mỗi repo build riêng.

⚡ Xung đột git

Conflict thật rất hiếm (mỗi vai làm trên file riêng). Nếu xảy ra → nhờ Dev resolve, không tự sửa.
